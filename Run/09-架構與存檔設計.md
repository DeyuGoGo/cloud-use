# 09 — 程式架構與系統底板

> 給接手的人／另一個 AI。這份**只談系統面**：故事參數、回憶、存檔、滑卡引擎——做一個**好擴展、與劇本無關**的底板。
> ⚠️ **本文不預設任何後續劇情。** 結局有幾個、誰跟誰、怎麼降溫、第幾章發生什麼……都是劇本的事，由寫手日後定。**引擎只提供「機制」，不決定「內容」。**
> 目前聚焦：**把開頭做穩 ＋ 打基礎建設 ＋ 玩法雛型**。對應現況見 `game/`（autoload `RunState`/`SceneRouter`、序章 `Opening.gd`+`CardRunner.gd`、日常 `MainRoom.gd`、手機 `PhoneScreen`/`AliveWall`/`ChatList`/`ChatRoom`）。缺的美術另見 [08-美術工單 第二期](08-美術工單.md)。

## 三個核心決定
1. **存檔切兩層 ＋ 多手動格**〔已拍板〕：Profile（跨存檔的收集）＋ Run（每格一份的單局進度）。
2. **不導入 Ink**，維持 GDScript 資料驅動（理由見五、）。
3. **一切內容都是資料、引擎不認劇情**：故事參數是**開放 key-value**、滑卡走**通用的條件/效果引擎**、回憶是**資料驅動的收集**——**加新參數／新角色／新事件／新結局都只改資料，不改引擎。** 這就是「好擴展的底板」。

## 一、存檔底板（純基礎建設）

### 切兩層（已拍板多手動格，但仍要分層）
不分層的話，`New Game`／刪某一格會把**圖鑑/回憶/結局收集**一起清掉——這些是跨存檔的東西。所以「跨周目的收集」與「單局的進度」必須分開。

| 層 | 檔案 | 放什麼 | 生命週期 | 寫入 |
|----|------|--------|---------|------|
| **Profile（全域，唯一份）** | `user://profile.json` | 設定（音量/震動/語速/語言）、**圖鑑/回憶解鎖（集合）**、**看過的結局（集合，由內容定義）**、玩家層級旗標、18+ 確認、累計統計、`last_played_slot` | 永久；跨所有存檔格、`New Game`／刪格都不清 | 事件觸發：解鎖回憶/看到結局/改設定。**atomic** |
| **Run（每格一份）** | `user://saves/slot_<n>/run.json` ＋ `meta.json` | `params{}`/`bond{}`/`flags{}`/`route`、`day`/`energy`/`money`/`week`/`phase`、**`chapter`/`beat` 游標**、**`seen_storylets{}`**、社交層狀態、history | 每格獨立；只有該格 `New Game`／刪格才清 | 自動存 slot 0；手動存 1..N。**atomic** |
| **Transient（不存）** | — | 卡片拖曳/選取、版面節點、F1 overlay、截圖路徑 | 當下 | 不寫 |

### 存檔格佈局（給讀檔選單）
- 一格一資料夾：`user://saves/slot_<n>/run.json` ＋同層 `meta.json`＝`{version, chapter/beat, day, leaning/route, saved_at, thumb:"thumb.png"}`。
- 讀檔選單**只讀各格 `meta.json`**（便宜），不解析完整存檔；縮圖存小 PNG、`meta` 只記路徑（別把圖塞進 JSON）。
- **slot 0 ＝自動存檔格**（每日行動/beat 邊界自動寫）；**1..N ＝手動格**。`繼續遊戲`＝讀 `last_played_slot`；主畫面「存檔」鈕＝寫一個手動格（含縮圖）。
- 序章滑卡**中途不自動存**，只在乾淨邊界存，避免存到半解析的卡。
- 之後若想要主題純粹版，可加**選用「承諾模式 / ironman」**開關（關手動存、單一自動格）放 Profile 設定、預設關——兩種模式共用同一存檔層。

### 存檔技術規格（多為現有 `RunState` 的小改）
| 項目 | 現況 | 目標 | 工 |
|------|------|------|----|
| 格式 | JSON（`user://run_save.json`） | **維持 JSON**（全是 scalar/dict/array，可讀、不能執行內嵌腳本）。**絕不**用 `.tres/.res` 當玩家存檔（Resource 可挾帶腳本＝18+ Steam 的 RCE/作弊洞）；`.tres` 只給出貨內建的設計資料。 | — |
| 版本 | 無 `version` 欄 | 每格 `run.json`＋`meta.json` 寫 `SAVE_VERSION`；load 時 `v < CURRENT` 走 `_migrate()` 階梯升級再存回。現有「reset 後逐鍵帶預設值」是 additive 變更的安全網。**公開版存檔前就要有。** | 小 |
| 寫入安全 | `FileAccess.open(WRITE)` 會**先截斷**真檔 | **atomic**：寫 `.tmp` → `flush()/close()` → 舊檔轉 `.bak` → `DirAccess.rename_absolute()`。load 失敗退 `.bak`。（Windows：rename 前關 handle、檢查回傳值） | 小 |
| 分層 | 全擠一個檔 | 拆出 `Profile` autoload；`RunState.reset()` **不准碰** profile。 | 中 |
| 多格 | 單一固定路徑 | 改 `user://saves/slot_<n>/`；`SaveManager.list_slots()/save_to(n)/load_from(n)`。`SceneRouter.continue_game()/new_game()` 是僅有呼叫點。需要一個**讀檔/存檔選單 UI**。 | 中 |

## 二、故事參數系統（故事參數）— 開放、可擴展

> 目標：寫手要新增一個參數/旗標/角色關係，**只在內容裡寫個名字就能用，不必改引擎**。

三種狀態，全部是**開放 key-value**：
- **`params{}`** — 方向性隱藏數值（目前 `seen/clean/nerve/self_first`，但**引擎不限定 key**；`RunState.apply()` 已是任意 key 累加）。永遠不顯示數字。
- **`bond{}`** — 對某對象的關係溫度，**key = 角色 id**。加一個角色＝多一個 key，不改碼。
- **`flags{}`**〔新增〕 — 具名故事旗標（bool 或計數），記「發生過什麼/做過什麼選擇」，給條件判斷與回憶用。

設計要點：
- **單一 mutation 原語**：把現有 `RunState.apply(fx)` 擴成 `Effects.apply(state, fx)`，支援 `params`/`bond`/`flags` 的增減與 clamp。**改狀態只有這一個入口。**
- **衍生讀法用「帶」不用裸數字**：`leaning()` 已示範把隱藏值轉成 band（seen/clean/middle）；一般化成 `band(key, thresholds)`，內容靠「帶」判斷語氣/選項，符合「不顯示數字、靠感覺」鐵律。
- **可選 registry（純文件）**：列出目前在用的 param/flag/bond key ＋一句說明，給 F1 overlay 與寫手查。engine 仍當開放字典處理，registry 缺了也能跑——它只是「有哪些參數」的清單，不是限制。

## 三、滑卡引擎（Storylet ＋ 條件/效果）— 通用機制，不認劇情

現有 `Opening.scenes()+FLOW` 已經是對的形狀的 80%。一般化成 **storylet**：
```
storylet = { id, title, loc, sprite, beats[],
             requires:{}, effects:[], weight, priority, once, cooldown, tags[], next }
# beat = {say} 或 {choice}
# choice side = { label, sub, icon, requires:{}, fx:{}, route, end, fb, goto, memory? }
```
- 把後果（`fx/route/end/fb`）從 `FLOW` 表搬到**每個選項上**，並給選項加 `requires`（依狀態顯示/反灰）。線性章節（`next` 串接）與事件池（Selector 依狀態挑）是**同一型別**。
- **唯一要新寫的引擎 ~120 行**：`Cond.eval(state, req)`、`Effects.apply(state, fx)`、`Selector.pick(pool, state)`。

引擎只提供「能力」，**用不用、怎麼用由劇本定，本文一律不預設內容**：
- 依狀態 gate/顯示/隱藏選項（「選項依感覺出現/消失」）。
- 小幅 `bond` delta ＋ 帶切換 → 可做「語氣慢慢變」（**這只是個能力，不是寫死的某段降溫劇情**）。
- **結局解析**：提供一個**通用的 ordered first-match 規則表**（`requires → 結局 id`，第一個 match 取勝）。**幾個結局、各是什麼、條件，全部留空給劇本**；引擎只負責「跑到收束點時依規則挑一個」。目前序章只有 3 個收束（home/lurk/post），那是開頭用的，不代表全遊戲的結局數。
- **對象翻面**：同一批 storylet 可重新指向不同 bond key——能不能拿來做「某角色換邊」之類，純資料，引擎不變。

內容放哪：**GDScript 一章一檔**（`Opening.gd` 目前是唯一一章，當引擎回歸基準）；`ChapterRegistry`/`EventPool` 匯總餵 Selector。先不 JSON、不 Ink。
⚠️ **上線玩法雛型前**，把 `flags`、`seen_storylets`、`chapter/beat` 加進存檔 schema，否則重玩會重觸發一次性事件。

## 四、回憶系統（回憶）— 資料驅動的收集與重溫

> 對應 07-介面規格「圖鑑：角色・場景・**回憶**」。同樣**不預設哪些算回憶**——寫手標，引擎收。

- **回憶 ＝ 一個被標記為「值得留下」的時刻**：某個 storylet/beat，或某個關鍵選擇的快照。
- **怎麼產生**：內容在 beat/選項上掛 `memory:{id, title}`，或用效果 `unlock_memory(id)`。寫手標哪些，引擎就收哪些。
- **存哪**：解鎖紀錄放 **Profile**（跨周目收集，「重玩仍然算數」）；要記「這一局的回憶」也可在 Run 放一份。一筆回憶可順手存**當時的參數快照/做了什麼選擇**，重溫時還原「當時的感覺」而非顯示數字。
- **怎麼重溫**：圖鑑回憶分頁列已解鎖；點開＝用 storylet 引擎**唯讀重播**那一拍（沿用同一個 `CardRunner` 渲染），或一張靜圖＋當時的話。
- **為什麼好擴展**：回憶不是另寫一套，而是「storylet ＋ 一個解鎖旗標 ＋ Profile 收集」——和故事參數、滑卡引擎**共用同一批機制**。

## 五、不導入 Ink（決定，精簡）
- 本專案是**標準（非 .NET）Godot 4.6**：`GodotInk`（C#）需 .NET 引擎＝得換整個引擎風味，出局；唯一相容的 `inkgd` 其 Godot 4 支援在**未釋出分支**、且較慢。
- 「不顯示數字」讓分支由 Godot 端 `bond`/band 驅動，Ink 變數只會變成**要同步的影子**，多一層 desync bug。
- 已寫好的 13 卡移植 `.ink` 是**沉沒成本＋負收益**。doc 06 §7 的「Godot+Ink」是從未實作的意圖、不是需求。
- 後路（只寫不做）：若日後有專職寫手且 dict 變瓶頸，才考慮 `inkgd` hybrid（Ink 管對白、Godot 握隱藏參數＋存檔）。

| **回憶系統**（解鎖＋Profile 收集＋唯讀重播） | 圖鑑「回憶」分頁；跨周目收集 | ✔ | 基礎建設 |
| **資料驅動社交層**（手機讀 `bond` 帶） | 07 鐵律：關係**只**從手機讀；現在 AliveWall/Chat 是寫死 const、與 bond 無關 | ✔ | 玩法雛型 |

## 七、建議的檔案 / autoload 佈局
```
autoload:
  RunState    既有，瘦成「當前 run 的 params/bond/flags ＋ mutation 原語」
  Profile     新增：profile.json（設定/圖鑑/回憶/結局集合/旗標），atomic
  SaveManager 新增：包 atomic 寫入＋備份＋version/migration＋多格；Run 與 Profile 都走它
  SceneRouter 既有，加 RUN/SYNC 場景常數與讀檔/存檔選單入口
scripts/sys/
  Cond.gd / Effects.gd / Selector.gd   滑卡引擎（class_name）
  ChapterRegistry.gd / EventPool.gd     匯總各章 storylets 餵 Selector
  MemoryBook.gd                         回憶解鎖/查詢/重播（讀 Profile）
scripts/story/

```

## 八、路線圖（聚焦開頭＋底板＋雛型）
1. **存檔底板**：拆 `Profile`、加 `version`/migration、atomic+backup、多格佈局、把 `flags/seen_storylets/chapter/beat` 進 schema。**不動玩法，只改存檔形狀。**
2. **故事參數系統**：開放 `params/flags/bond` ＋ `Effects.apply` ＋ `band()`，F1 overlay 顯示。
3. **滑卡引擎**：`Cond/Effects/Selector` ＋ 把 `Opening.gd` 重構成 storylets（**行為不變、當回歸基準**）＝把開頭跑在新底板上。
4. **回憶系統**：在開頭挑幾個時刻當 demo，驗「解鎖→圖鑑→重溫」。
5. **資料驅動社交層**（手機讀 bond 帶）＋ **讀檔選單 UI** ＋ **設定/音訊**。
→ 到這裡：**開頭穩、底板可擴、玩法雛型成立**。後續劇情/結局/同步/中後期，等你劇本想好，直接用這套底板長，不必再改引擎。
